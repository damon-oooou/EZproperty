# v0.6 阶段 C 验收清单 — 云端部署

改动文件：`Dockerfile`、`.dockerignore`、`application-prod.yml`、
`DatabaseUrlEnvironmentPostProcessor.java` + `META-INF/spring.factories`（DATABASE_URL 映射）、
`CorsConfig.java`（origins 配置驱动）、`frontend/src/api/client.js`（VITE_API_BASE_URL）、
`frontend/src/pages/RoomPage.jsx`、`frontend/vercel.json`（SPA rewrite）、`frontend/.env.example`。

部署顺序：数据库 → 后端 → 前端 → 域名。

## 1. 本地预检（部署前）

```powershell
# Docker 镜像能构建、prod profile 能起(复用阶段 B 的 R2 环境变量)
docker build -t ezproperty-api .
docker run --rm -p 8080:8080 `
  -e SPRING_PROFILES_ACTIVE=prod `
  -e DATABASE_URL="postgresql://postgres:postgres@host.docker.internal:5432/property_photos" `
  -e JWT_SECRET="<随机长串>" `
  -e R2_ENDPOINT=$env:R2_ENDPOINT -e R2_ACCESS_KEY=$env:R2_ACCESS_KEY `
  -e R2_SECRET_KEY=$env:R2_SECRET_KEY -e R2_BUCKET=$env:R2_BUCKET `
  -e CORS_ALLOWED_ORIGINS="http://localhost:5173" `
  ezproperty-api
# 期望:启动成功,日志显示 DATABASE_URL 被映射(datasource 连上本地 PG)
Invoke-RestMethod -Uri "http://localhost:8080/actuator/health"   # {"status":"UP"}
```

## 2. Railway（所有者操作）

1. New Project → 区域 **Singapore** → 添加 **PostgreSQL** 插件。
2. 添加后端服务：连 GitHub repo，自动识别 `Dockerfile`。
3. 后端服务 Variables：

| 变量 | 值 |
|---|---|
| `SPRING_PROFILES_ACTIVE` | `prod` |
| `DATABASE_URL` | 引用 PG 插件：`${{Postgres.DATABASE_URL}}` |
| `JWT_SECRET` | 新生成 ≥64 字节随机串（生成：`[Convert]::ToBase64String((1..64 \| %{Get-Random -Max 256}))`） |
| `GOOGLE_CLIENT_ID` | 现有 OAuth Client ID |
| `R2_ENDPOINT` / `R2_ACCESS_KEY` / `R2_SECRET_KEY` / `R2_BUCKET` | 生产桶 `ezproperty-prod` 的凭证 |
| `CORS_ALLOWED_ORIGINS` | 先填 Vercel 生成域名（下一步拿到后回来填） |

4. Settings → Networking → Generate Domain，记下 `https://xxx.up.railway.app`。
5. 部署日志确认：Flyway `V1 → V11` 全部成功、`Started PropertyPhotosApplication`。

## 3. Vercel（所有者操作）

1. Import GitHub repo → Root Directory 选 `frontend/`（Vite 自动识别）。
2. 环境变量：`VITE_API_BASE_URL = https://xxx.up.railway.app`（不含 /api）、
   `VITE_GOOGLE_CLIENT_ID = <client id>`。
3. Deploy，记下 `https://xxx.vercel.app` → 回填 Railway 的 `CORS_ALLOWED_ORIGINS`（后端会自动重启）。
4. Google Cloud Console → OAuth Client → Authorized JavaScript origins 加 `https://xxx.vercel.app`
   （临时，正式域名后替换）。

## 4. 域名（Cloudflare Registrar）

1. 购买域名。
2. DNS：`app` CNAME → `cname.vercel-dns.com`（Vercel 项目 Domains 页会给准确目标）；
   `api` CNAME → Railway 服务 Custom Domain 页给的目标。两侧平台自动签证书。
3. 生效后回改三处：
   - Railway `CORS_ALLOWED_ORIGINS` → `https://app.<domain>`
   - Google OAuth Authorized JavaScript origins → 加 `https://app.<domain>`
   - Vercel `VITE_API_BASE_URL` → `https://api.<domain>` 并 Redeploy

## 5. 验收项

1. **手机 4G**（不连开发机网络）访问 `https://app.<domain>`：
   注册 → 登录 → 建 property → 建 inspection → 传照片 → 编辑报告 → 下载 PDF，全流程通。
2. Google 登录在正式域名下正常。
3. 照片三档均经 R2 签名 URL 显示（DevTools 确认 URL 含 `X-Amz-Signature`）。
4. 安全检查：

```powershell
# 无 token → 401
Invoke-WebRequest -Uri "https://api.<domain>/api/properties" -SkipHttpErrorCheck |
  Select-Object StatusCode
# 非白名单 origin 的预检 → 无 Access-Control-Allow-Origin 头
Invoke-WebRequest -Method Options -Uri "https://api.<domain>/api/properties" -SkipHttpErrorCheck `
  -Headers @{ Origin = "https://evil.example"; "Access-Control-Request-Method" = "GET" } |
  Select-Object -ExpandProperty Headers
```

5. Railway 日志确认 Flyway V1→V11 全部成功。
6. 本地 dev 完全不受影响：`docker compose up -d; mvn spring-boot:run`（无 profile）照 README 跑通。
