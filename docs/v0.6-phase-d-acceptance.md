## 2. R2 防误删

**⚠️ 受阻于平台功能缺失（2026-07-14 核实）：** R2 目前不支持 Object Versioning，
该功能在 Cloudflare roadmap 上但无发布日期——规格书此项基于错误假设，无法按原文执行。

**所有者已拍板（2026-07-14）：记 backlog，等 R2 versioning 上线后开启 + 非当前版本 30 天清理。**
残余风险评估：照片在应用流程中几乎不可变（正常业务只删引用不删对象，
对象删除仅存在于上传失败回滚清理），误删面很小；R2 自身持久性 11 个 9。
已否决的替代项：Bucket Lock（会阻断上传回滚清理，堵死将来的 library 删除）、
第二桶定期同步（现阶段过度设计）。

## 3. Auth 限流

```powershell
# login:11 次错误密码,第 11 次应 429(前 10 次 401)
1..11 | ForEach-Object {
  $r = Invoke-WebRequest -Method Post -Uri "https://api.<domain>/api/auth/login" `
    -ContentType "application/json" -Body '{"email":"x@x.com","password":"wrong"}' -SkipHttpErrorCheck
  "$_ -> $($r.StatusCode)"
}

# X-Forwarded-For 验证点(本地 dev 直测 filter 逻辑:两个伪造 IP 分别计数)
1..11 | ForEach-Object {
  $r = Invoke-WebRequest -Method Post -Uri "http://localhost:8080/api/auth/login" `
    -Headers @{ "X-Forwarded-For" = "1.1.1.1" } `
    -ContentType "application/json" -Body '{"email":"x@x.com","password":"wrong"}' -SkipHttpErrorCheck
  "ip1 $_ -> $($r.StatusCode)"   # 第 11 次 429
}
$r = Invoke-WebRequest -Method Post -Uri "http://localhost:8080/api/auth/login" `
  -Headers @{ "X-Forwarded-For" = "2.2.2.2" } `
  -ContentType "application/json" -Body '{"email":"x@x.com","password":"wrong"}' -SkipHttpErrorCheck
"ip2 -> $($r.StatusCode)"        # 401,不受 ip1 影响

# register:每 IP 5 次/小时,第 6 次 429(注意会真的创建账号,用一次性邮箱)
```

**验收：** 超限 429 且消息友好；正常登录不受影响；两个 IP 独立计数。
**待部署后核实（规格书要求）：** 在 Railway 上确认 X-Forwarded-For 第一跳确为客户端真实 IP
（部署后从两台不同网络设备各打一次，比对 Railway 日志/临时打印的 clientIp）。

登录报错核对：用户不存在与密码错误均返回 401 + "Invalid email or password"（`AuthService.login`
已是单一 `orElseThrow`，代码审查通过）。

## 4. 监控

**Sentry（所有者操作）：** sentry.io 建两个项目（backend Java / frontend React），拿两个 DSN：
- Railway 后端服务加环境变量 `SENTRY_DSN`；

**验收：** 前端在浏览器 console 执行一次
`window.dispatchEvent(new ErrorEvent('error', { error: new Error('sentry-fe-test') }))`
（或临时加一个抛错按钮）→ Sentry 前端项目收到事件。
后端临时触发一个未处理异常（例如临时在任意 controller 加一行 `throw new RuntimeException("sentry-be-test")`
访问一次后撤掉）→ 后端项目收到事件。

**UptimeRobot（所有者操作）：** HTTP monitor 指向 `https://api.<domain>/actuator/health`，
间隔 5 分钟，告警邮箱。**验收：** 暂停 Railway 服务几分钟（或用 UptimeRobot 测试功能）→ 收到宕机邮件。

Actuator 已收紧：`management.endpoints.web.exposure.include=health`，
`/actuator/health` 之外的端点均不暴露（可用 `Invoke-WebRequest https://api.<domain>/actuator/env -SkipHttpErrorCheck` 验证 404/401）。

## 5. 密钥卫生审计（已完成，结论如下）

扫描范围：全部 14 个提交的 yml/properties/env 文件 + 长随机串模式扫描 + `.env` 类文件入库记录。

| 发现 | 定性 | 处置 |
|---|---|---|
| `docker-compose.yml` / `application.yml` 中 `postgres/postgres` | 仅本地 Docker dev 库，不对外 | 无需处理 |
| `application.yml` JWT dev 默认值 | 规格书明确豁免（仅 dev profile） | 无需处理；prod 无默认值，缺 `JWT_SECRET` 启动即失败 |
| Google OAuth **Client ID**（application.yml 默认值 + 历史中的 .env.example） | Client ID 是公开标识符（本就随前端 JS 下发），非密钥 | 无需轮换；`.env.example` 中已清空该值 |
| `.env.local` 及任何真实凭证 | 从未入库 | — |

`.gitignore` 核对：根目录覆盖 `.env`、`uploads/`、`frontend/.env`；`frontend/.gitignore` 的
`*.local` 覆盖 `.env.local` ✓。**结论：git 历史无需轮换任何凭证。**
