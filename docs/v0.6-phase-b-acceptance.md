## 0. 所有者手动准备（Cloudflare 控制台）

1. 建桶 `ezproperty-prod`（联调可另建 `ezproperty-dev`），保持私有（默认即私有，不开公共访问）。
2. R2 → Manage API Tokens → 建 Token（Object Read & Write，限定桶），记下 Access Key ID / Secret。
3. 记下 S3 endpoint：`https://<account_id>.r2.cloudflarestorage.com`。

## 1. prod profile 本地起后端（连 R2）

```powershell
$env:SPRING_PROFILES_ACTIVE = "prod"
$env:R2_ENDPOINT   = "https://<account_id>.r2.cloudflarestorage.com"
$env:R2_ACCESS_KEY = "<access-key>"
$env:R2_SECRET_KEY = "<secret-key>"
$env:R2_BUCKET     = "ezproperty-dev"   # 联调桶
$env:JWT_SECRET    = "<随便一个长随机串>"
mvn spring-boot:run
```

> 注意：此时 application-prod.yml 尚不存在（阶段 C 创建），datasource 沿用
> application.yml 的 localhost 配置——正好符合"后端仍在本地"的阶段目标。

## 2. 验收项

1. 上传一张照片 → Cloudflare 控制台桶内出现三个对象：`{uuid}.jpg` / `{uuid}_m.jpg` / `{uuid}_t.jpg`。

2. 照片接口返回的三个 URL 均为 `https://<account_id>.r2.cloudflarestorage.com/...X-Amz-Signature=...` 形态；网格/lightbox/原图正常显示：

```powershell
Invoke-RestMethod -Uri "http://localhost:8080/api/inspections/$inspId/rooms/$roomId/photos" -Headers $H |
  Select-Object thumbnailUrl, mediumUrl, originalUrl | Format-List
```

3. 裸 object URL（去掉签名参数）直接访问 → 401/403：

```powershell
# 取 originalUrl 问号前的部分
$bare = ($photos[0].originalUrl -split '\?')[0]
Invoke-WebRequest -Uri $bare -SkipHttpErrorCheck | Select-Object StatusCode   # 期望 401/403
```

4. 篡改签名 → 拒绝：

```powershell
$tampered = $photos[0].originalUrl -replace 'X-Amz-Signature=.{8}', 'X-Amz-Signature=deadbeef'
Invoke-WebRequest -Uri $tampered -SkipHttpErrorCheck | Select-Object StatusCode   # 期望 403
```

刷新页面重新拿 URL → 正常。

5. PDF 生成正常、照片方向正确；观察日志/耗时确认多张图时为并行拉取（数十张图应在数秒内）。

6. 删除照片引用（Remove from inspection）行为不变——只删 join 行，桶内对象不动。
   > 记录：当前产品没有"从 library 彻底删除照片"的端点（photos 上是 RESTRICT 约束），
   > `PhotoStorage.delete()` 已实现三档一并删除，供上传回滚清理使用，并为将来的
   > library 删除功能预留。规格书验收项 5 的后半句（桶内对象消失）暂无 API 可触发，
   > 已记 backlog 向所有者提出。

7. 切回 dev：`Remove-Item Env:SPRING_PROFILES_ACTIVE` 后 `mvn spring-boot:run`，
   一切照旧（本地 uploads、/uploads/** 可访问、无需 R2 凭证、不触碰网络）。
